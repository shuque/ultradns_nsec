# Distance Between DNS Domain Names

## Problem Statement

To detect minimally covering NSEC records (RFC 4470) — also known as
"white lies" or epsilon functions — it would be useful to compute a
**distance metric** between two DNS domain names in canonical order
(RFC 4034 Section 6.1). An NSEC record asserts that no name exists
between its owner name and its "next" name. If the distance between
these two names is very small, it is likely that the NSEC was
synthetically generated by an online signer using an epsilon function
rather than being part of a pre-signed static NSEC chain.

A reliable distance metric would allow detection of epsilon functions
from various DNS providers (UltraDNS, Cloudflare, etc.) without needing
to reverse-engineer each provider's specific algorithm. Instead, we could
simply measure the distance in each NSEC record and flag suspiciously
small values.

### Challenges

DNS canonical ordering is tree-based, not flat lexicographic. Children of
a name sort between that name and its next sibling:

```
a.zone < child.a.zone < b.zone
```

Any distance metric must respect this hierarchical structure. Additionally,
DNS names can be up to 255 bytes long with labels up to 63 bytes each,
creating an enormous namespace that the metric must span.

## Approaches Considered

### Option 1: Canonical Sort Key as Big Integer

Construct a byte string for each name (relative to the zone) that, when
compared lexicographically, preserves DNS canonical ordering. Then
interpret both keys as big-endian integers and subtract.

The encoding uses length-prefixed labels ordered from the zone outward:

```
a.zone       -> \x01a\x00
child.a.zone -> \x01a\x05child\x00
b.zone       -> \x01b\x00
```

The `\x00` end-of-name marker sorts before any length byte (1-63), so a
parent always sorts before its children. Children sort before the next
sibling because the first label already decided the comparison.

Both keys are padded to the maximum DNS name length (255 bytes) with
`\x00` and interpreted as big-endian integers. The distance is the
absolute difference. Python handles arbitrary-precision integers natively
(up to 2^2040 for 255-byte keys).

### Option 2: Fractional Position in \[0, 1)

Same sort key as Option 1, but interpret the bytes as a binary fraction.
Mathematically equivalent with different presentation.

### Option 3: Label-Level Distance (Heuristic)

For each NSEC pair, find the common ancestor, then measure the distance
only at the first diverging label level — treating that label as a
base-256 number. Ignore deeper structure entirely.

## Why Raw Distance Alone Fails

Prototyping revealed that **raw distance between NSEC owner and next
name does not reliably distinguish epsilon from static NSECs**, regardless
of whether the full canonical sort key (Option 1) or label-level distance
(Option 3) is used.

The fundamental issue: epsilon predecessors use child labels (e.g.,
`~.koal_~`) which are canonically close to the query name but numerically
span the entire sub-namespace under the base label. At the label level,
the distance between `koal_~` and `koala!` is comparable to the distance
between arbitrary real zone names like `address1` and `address2`. The
"tightness" of the epsilon bracket is encoded in the **structure** of the
names (child labels, alphabet-edge characters), not in a scalar distance.

Tested distances (label-level, log2 scale, max = 504 bits):

| Pair | Type | Distance (bits) |
|---|---|---|
| `~.koal_~` → `koala!` | epsilon | 464.7 |
| `~.abcdee~` → `abcdef!` | epsilon | 455.3 |
| `~.l~` → `m!` | epsilon | 495.3 |
| `address1` → `address2` | static | 440.0 |
| `a` → `b` | static | 496.0 |
| `jaguar` → `panthro` | static | 499.9 |

The ranges overlap completely — `address1` → `address2` (static, 440
bits) looks "tighter" than `~.l~` → `m!` (epsilon, 495 bits).

## What Works: Prefix Similarity to Query Name

The key insight is that epsilon NSECs are **derived from the query name**,
while static NSECs are pre-existing zone names with no structural
relationship to the query. Instead of measuring distance between the NSEC
endpoints, we measure how closely the NSEC labels match the **queried
label**:

- **Successor (NSEC next)**: For epsilon, next = qname + suffix (e.g.,
  `koala!`), so the query label is always a prefix of the next label.
  For static, the next label is a real zone name unrelated to the query.

- **Predecessor (NSEC owner)**: For epsilon, the outermost label shares
  all but the last 1-2 bytes with the query label (e.g., `koal_~` shares
  `koal` with `koala`). For static, the owner label is unrelated.

The metric: **prefix match ratio** = (shared prefix bytes between query
label and NSEC label) / (query label length).

## Tool: dns\_name\_distance.py

The implementation provides two subcommands:

### calc — Analyze a single NSEC pair

Given an NSEC owner, NSEC next, and optionally the query name that
produced the NSEC, computes the label-level distance and prefix
similarity.

```
Usage: detect_minimal_nsec.py calc ZONE NAME1 NAME2 [--qname NAME]
```

Without `--qname`, only the label distance is reported (which, as noted
above, is insufficient to distinguish epsilon from static). With
`--qname`, the prefix similarity analysis is enabled and a determination
is made.

**Example — epsilon NSEC:**

```
$ python3 detect_minimal_nsec.py calc ultratest.huque.com '~.koal_~' 'koala!' --qname koala

Zone:             ultratest.huque.com.
NSEC owner:       ~.koal_~.ultratest.huque.com.
NSEC next:        koala!.ultratest.huque.com.
Closest encloser: ultratest.huque.com.
Diverging labels: koal_~ vs koala!
Label distance:   2^464.7 (464.71 / 504 bits)

Query name:       koala.ultratest.huque.com.
Query label:      koala
Prefix match:     owner=4/5 (80%)  next=5/5 (100%)

  => NSEC boundaries track the query name
     Consistent with minimally covering NSEC (epsilon function)
```

**Example — static NSEC:**

```
$ python3 detect_minimal_nsec.py calc nseczone.huque.com jaguar sub1 --qname nhgwiwcu

Zone:             nseczone.huque.com.
NSEC owner:       jaguar.nseczone.huque.com.
NSEC next:        sub1.nseczone.huque.com.
Closest encloser: nseczone.huque.com.
Diverging labels: jaguar vs sub1
Label distance:   2^499.2 (499.18 / 504 bits)

Query name:       nhgwiwcu.nseczone.huque.com.
Query label:      nhgwiwcu
Prefix match:     owner=0/8 (0%)  next=0/8 (0%)

  => NSEC boundaries do not track the query name
     Consistent with static/pre-signed NSEC chain
```

### probe — Automatically query a zone and analyze NSEC patterns

Sends random queries to a zone, extracts the name-covering NSEC from
each response (filtering out wildcard-coverage NSECs), and reports the
prefix similarity for each. Produces an aggregate determination.

```
Usage: detect_minimal_nsec.py probe ZONE [-n NUM] [--doh] [--doh-server URL]
```

| Zone | Type | Avg owner prefix | Avg next prefix | Min next prefix |
|---|---|---|---|---|

The separation is definitive. For epsilon zones, every NSEC next label
starts with the full query label (100% match). For static zones, the
match is near zero because real zone names have no structural relationship
to random queries.

## Detection Algorithm

```
For each of N random queries:
  1. Send query for random_label.zone A with DNSSEC
  2. Find the NSEC in the response whose next label best matches
     the query label (this filters out wildcard-coverage NSECs)
  3. Compute prefix_match(query_label, next_label) / len(query_label)

If min(next_prefix_ratio) >= 0.75 across all probes:
  → Minimally covering NSEC (epsilon function) detected
Otherwise:
  → Static/pre-signed NSEC chain
```

The next prefix ratio is the strongest signal. The owner prefix ratio
and label distance provide supporting evidence. The consistency across
multiple probes (every query must show the pattern) eliminates false
positives from coincidental prefix matches with real zone names.

## Conclusion

Pure distance metrics are insufficient to distinguish epsilon from static
NSECs due to the tree structure of DNS canonical ordering. The effective
approach is **prefix similarity analysis** between the NSEC labels and
the query name. Epsilon functions construct NSEC boundaries that mirror
the query name; static NSEC chains use pre-existing zone names that bear
no relation to the query. This signal is robust, requires no knowledge of
the specific epsilon algorithm, and produces definitive separation across
all tested zones.
